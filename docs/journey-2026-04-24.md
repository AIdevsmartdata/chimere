# The Chimère Journey

*A local-first inference runtime for a 35-billion-parameter hybrid SSM+MoE
model, written by one person on a single 16 GB consumer GPU. Honest account
of two months of commits, pivots, and dead ends.*

*Author: Kévin Rémondière, April 24, 2026.*

---

## 1. Pitch

Chimère exists because I want a capable assistant that runs on my own
hardware, today, with no cloud dependency and no multi-GPU setup. It is
built by one person (me) plus a rotating set of Claude and Qwen agents,
on an Intel i5-14600KF / RTX 5060 Ti 16 GB / 32 GB DDR5 desktop. As of
2026-04-24 the production stack serves Qwen3.6-35B-A3B UD-IQ3_S at about
94 tok/s on a single 16 GB GPU with a 4-slot native scheduler, a
Prometheus-instrumented Rust server, and an OpenAI-compatible HTTP API
behind a Python orchestrator (ODO) and a Tauri desktop UI.

## 2. The repos

The family is five public repos plus one fork. They only became public
around March 29, 2026, but the work they contain started several months
earlier in a private mono-repo:

- **`chimere`** (<https://github.com/AIdevsmartdata/chimere>). The Rust
  runtime. ~96 commits on `main` since the March 29 open-sourcing, plus
  14 feature branches (`m1-multislot-j1`, `m2-j2-wiring`,
  `polish-prometheus`, `tq3-gguf-loader-support`, etc.). This is where
  `chimere-server`, the FFI glue to `ik_llama.cpp`, the `NativeScheduler`
  multi-slot path, the Engram n-gram bias and the `chimere-dflash`
  submodule all live.
- **`chimere-odo`** (<https://github.com/AIdevsmartdata/chimere-odo>).
  Python orchestrator. Single-process proxy in front of `chimere-server`
  that does intent routing, deep-search, quality gates, Pre-Act planning,
  XGrammar-2 constrained decoding, and exposes an MCP server so that
  Chimère Studio can drive RAG / engram / deep-search / OCR as tools.
- **`chimere-studio`** (<https://github.com/AIdevsmartdata/chimere-studio>).
  Tauri 2 + Next.js 14 desktop/mobile UI, scaffolded on April 24, 2026.
  Granola-style voice overlay, Pyodide/Mermaid/JS artifacts, Memory
  Inspector, chat with reasoning-content split.
- **`ramp-quant`** (<https://github.com/AIdevsmartdata/ramp-quant>).
  Quantization pipeline. RAMP v1 / v2 custom-q mix, imatrix calibration
  corpus, later the TQ3 variants.
- **`ik_llama.cpp`** fork
  (<https://github.com/AIdevsmartdata/ik_llama.cpp>). Forked from
  Iwan Kawrakow's fork of `llama.cpp`. We've been using this as the C++
  backend since day one, and eventually contributed the Mamba-2 +
  Nemotron-H backport upstream as PR #1593.

The `chimere-dflash` submodule, with the DFlash / EAGLE drafter code, is
vendored inside `chimere` rather than standalone. It represents a
research track that has been parked (section 5).

Throughout this document, commit SHAs are real and resolvable at the
time of writing.

## 3. The chronology

### 2026-01 to 2026-02 — Before the cameras

This period is not in `chimere/.git`. It lives inside the `OpenClaw`
agent framework and a private Rust workspace called `chimere-deltanet`.
The memory files `chimere-perf-sprint-2026-03-15.md` and
`chimere-session-2026-03-17.md` are the record. Highlights:

- First MoE Rust runtime with IQ3_S CUDA dequant on `sm_120` runs at
  **12.6 tok/s**. "Model loads in 6s (14.7 GB IQ3_S custom-mix, raw U8
  experts on GPU)."
- A performance sprint on March 15 takes the 35B-A3B from 0 → 21.1
  tok/s via LTO, GPU-resident state, transposed storage, fused
  sigmoid/silu/rms_norm, and batch prefill. The 27B dense variant of
  Chimère (non-MoE) goes from 9.1 → 20.5 tok/s in the same session.
- On March 16, session 3 pushes to **42.5 tok/s** with a GQA fused
  attention kernel (`CHIMERE_GQA_FUSED=1`) and a raw SSM path (2 tries,
  one heisenbug on `head % n_group` vs `head / repeats`).
- On March 17 a "marathon debug" fixes seven bugs at once: shared expert
  gate was computing `sigmoid(weight)` element-wise instead of
  `sigmoid(hidden @ weight)` scalar (one line × three call sites, turned
  garbage into coherent text); double-forward of last prompt token; L2
  norm `sqrt(x²+eps)` vs `max(√x², eps)`; f16 truncation via bit-shift
  instead of `cvt.rn.f16.f32`; softplus overflow; missing Q5_K dequant;
  and the sampling pipeline itself was not wired.
- Late March 17, the "missing layer 39" bug. GGUF custom-mix had pruned
  the MTP layer but kept `nextn_predict_layers=1` in metadata, so the
  runtime loaded only 39/40 layers. Symptom: `"2+2=?"` answered
  `"usercontentusercontentu..."` in a loop. Fix in `config.rs` and a hex
  patch on the GGUF at offset 1318.

By March 21 there is a tagged `chimere-prod-v1-final`: **96 tok/s** HTTP
non-streaming, 77-87 tok/s SSE with logprobs, C++ fast sampler, ABF
logprobs infrastructure, `<think>` tag extraction, tool calling in the
Qwen3.5 Jinja format, sequential 3-agent context switching. That is the
version we opened in public on March 29.

**What I would tell past-me:** the greedy divergence from `ik_llama` is
not a bug to fix by tuning. It's structural — Candle has no IQ3_S CUDA
kernel (CPU dequant + F32 matmul vs fused GPU IQ3_S+Q8_1), old DMMV
kernels, different FP32 accumulation order (1 warp vs 4 warps), and
those compound over 40 layers. Switch to the `ik_llama.cpp` FFI for the
quantised matmuls and keep cudarc for the GDN custom path.

### 2026-04-06 — The Mamba-2 backport and the gateless FFN bug

April 6 is the `ik_llama.cpp` Phase 3 backport. Starting from commit

one day the fork lands:

- `15c49825` — Phase 2: hparams loading + tensor allocation

- `ce543ad4` — Phase 3.3: `build_mamba` + `build_nemotron_h_moe`
- `ee0c4243` — `eval-callback` sum over full tensor
- `9bc3ee1c` — `inp_ssm_ids` reads recurrent slot 0, not
  `kv_self.head`
- `38cdc3b4` — **the bug**: `llm_build_ffn` had an unconditional
  `mul(cur, tmp)` in the `LLM_FFN_PAR` branch (lines 910 and 963 of
  `llama-build-context.cpp`). When `gate == nullptr` and the FFN was
  ReLU² gateless, this collapsed `relu²(up)` into `relu²(up) * up`. Every
  previous caller of `LLM_FFN_PAR` had passed a real gate, so the bug
  was unreachable until Nemotron-H's shared expert exercised it.

The memory file `ik-llama-iq3xxs-ssm-out-bug.md` documents the
diagnostic: instead of staring at audit-style reviews of
`build_mamba2_layer`, side-by-side `llama-eval-callback` traces between
`ik_llama` and upstream `llama.cpp` on the same GGUF, walking sums of
intermediate tensors layer-by-layer until the first divergence. Cheap,
conclusive, ~30 min. That's the technique to use first next time.

An earlier memory had blamed the `iqk` IQ3_XXS matmul kernel. That
diagnosis was wrong — sums looked like they correlated with quant type
but actually correlated with cumulative error across layers, and
Nemotron's alternating SSM / gateless-MoE layers compound fast.

Same evening, `8c4698f` in `chimere`: end-to-end Nemotron-3-Nano smoke
via `LlamaForward` FFI at 45 tok/s on `sm_120`, `NCMOE=30`. Later opens
the upstream PR [ikawrakow/ik_llama.cpp#1593](https://github.com/ikawrakow/ik_llama.cpp/pull/1593).

## 4. Pivots

### Pivot 1 — The production model: Qwen3.5 RAMP v3 → Qwen3.6-35B-A3B UD-IQ3_S

On March 29 the production model became `chimere-v3-ramp.gguf`: a 15.2
GB custom RAMP v2 mix (IQ3_S base + chimere imatrix + SSM Q8_0),
delivering 90 tok/s gen at 32K context on ik_llama `sm_120`
(`ramp-pipeline-validated-2026-03-29.md`). The mix was
tensor-by-tensor: `ssm_alpha`, `ssm_beta`, `ssm_out`, `attn_v`, `output`
all at Q8_0; `attn_gate/qkv/q`, `ffn_*_shexp` at Q5_K; `attn_k`,
`token_embd` at Q6_K; MoE experts left on the IQ3_S base. That stack
ran as the boot service for almost a month.

Between April 14 and April 24, a shift to **Qwen3.6-35B-A3B UD-IQ3_S**
occurred. The version goes from 3.5 → 3.6 (Qwen released 3.6, we took
it). The quant goes from a custom RAMP mix to Unsloth's Dynamic-IQ3_S.
The gen speed goes from 90 to 94 tok/s. It is also the version on
which the M1 multi-slot study was run. What tipped the scale:

1. The UD-IQ3_S weights ship 12.73 GiB at 3.15 BPW vs RAMP v2's 15.2 GB
   at 3.78 BPW. That's 2.5 GB of VRAM freed.
2. Unsloth's Dynamic mix is regenerated per release and we consume it
   directly, instead of maintaining a custom pipeline against each new
   arch variant.
3. With the M2 prefix cache landing the same week, the freed VRAM
   matters more than it did for single-slot.

The RAMP v1/v2 assets are preserved (section 7). The pipeline itself
is still valid, just not the first choice for this specific arch.

### Pivot 2 — Closing MTP and DFlash for this hardware

The `chimere-strategy-pivot-2026-03-19.md` memory reads verbatim:

> MTP et DFlash spec-decode sont FERMÉS. Le hardware (ncmoe=4 =
> experts CPU) rend le speculative decoding non-rentable quelle que
> soit l'acceptance rate.

The numbers that made that call:

- **MTP** (built-in Qwen3.5 MTP head, one extra layer with 256 MoE
  experts). Without `-mtp`: 94 tok/s. With `--draft-max 0`: 85 tok/s.
  With `--draft-max 1` (single-step): 47 tok/s with 85% acceptance hot
  cache, 61% cold cache. With `--draft-max 2+`: 12 tok/s and garbage
  output (MTP vanilla `k>1` collapses from 70% acceptance at k=1 to 11%
  at k=2 to 2% at k=3, per the FastMTP paper).
- **DFlash** (block diffusion drafter, 8-layer bidir Transformer + deep
  KV). Offline `τ=9.4` tokens/block, `+47%` over the published `τ≈6.4`
  from the z-lab paper. Online: the GDN State Barrier.

The GDN State Barrier is the insight worth its own memory
(`dflash-eagle-gdn-barrier.md`). Gated DeltaNet state is updated
sequentially per-token, and rolling back a rejected draft would
require `llama_memory_seq_rm(p0, p1)` — but the GDN state is monolithic
and cannot be trimmed to an intermediate position. Rolling checkpoints
per-token would be prohibitive in memory. Spec-decode cannot verify
online on any Qwen3.5 / Qwen3.6 / Qwen3.5-Coder-Next (all GDN hybrids).

The code does not disappear. It becomes an archived research paper:
`paper/02-dflash-moe-gdn-barrier-outline.md` + `paper/latex/dflash-moe-gdn.tex`.
The upside lives in the paper, not in our throughput.

**Do not re-propose DFlash or EAGLE as "quick wins" for Chimère
inference speedup.** (It appears in the CLAUDE.md as a standing
instruction.)

### Pivot 3 — TQ3 upstream closure

PR `ikawrakow/ik_llama.cpp#1683` was pushed on April 24 morning after
the Phase 1 CPU port and Phase 2 CUDA kernels landed in the fork. By
evening it was closed, on data that Ikawrakow ran himself:

| Quant | Size | PPL |
|---|---|---|
| TQ3_4S (ours) | 13.3 GiB | 6.93 |
| IQ3_KT (ikawrakow) | same | 6.02 (−13% vs TQ3) |
| **IQ2_KS + IQ3_KS mix** | **11.6 GiB** | **6.20 (−11% AND smaller)** |
| IQ2_KS + IQ3_KS light | 10.5 GiB | 6.35 |

The TurboQuant WHT + Lloyd-Max codebook does not match this model
family's weight distribution. It wins on other archs, and generalising
to Qwen3.6-35B-A3B was wishful. The meta-lesson is recorded in the
memory `tq3-upstream-closed.md`:

> Always compare against the upstream quants of the maintainer's own
> family BEFORE proposing a new method. ikawrakow's KS/KT series was
> the obvious baseline to check, not just the generic "llama.cpp
> IQ3_S" baseline we used.

The `tq3-gguf-loader-support` branch of `chimere` stays, in a
"historical reference" state, not intended for upstream. Production
stays on Unsloth UD-IQ3_S. The TQ3 detection code (`83da13b`) is
kept in the Rust GGUF parser — parsing cost is zero, it preserves
compat with any existing TQ3 GGUFs someone may already have.

### Pivot 4 — Cairn goes private and commercial

The `cairn-commercial-pivot-2026-04.md` memory is April 24 too. Cairn
is a separate project (FTMS Wear OS + self-improving local pipeline)

This matters to the Chimère journey because Cairn is the commercial

### Pivot 5 — From "make chimere fast" to "make chimere fair under load"

Not a memory-documented pivot, more a read I have of the April 24
session itself. Until April 7, every performance commit was about
peak single-slot tok/s (9 → 12 → 21 → 42 → 80 → 94). The April 24
session is almost entirely about **multi-slot fairness, TTFT under
load, and operator instrumentation** — `NativeScheduler`, J7 stress
test, Prometheus, `/v1/status`, profile hooks, prefix cache, sweep
harness, tuning guide, E2E benchmark, scheduling gap analysis.

The peak single-slot number (94 tok/s) didn't move much that day.
What moved was: TTFT p50 at M=4/PCH=512 from 747 to 422 ms (−43%);
four-slot cross-slot contamination proven zero via piège prompts;
per-slot decode collapse characterised honestly (it is 1/M, not better);
and the GDN batched-dispatch fix documented as a 2.9× upside available
in the fork. The product shifted from "fast for me" to "fair for two
to four concurrent users on the same box". That is a different game.

## 5. Hardware

### What the RTX 5060 Ti 16 GB enables

- `sm_120` (Blackwell consumer) with FP4 tensor cores but no WGMMA/TMA,
  no TMEM, no warp groups. Ampere-era `mma.sync` is the target, not
  Hopper guides.
- 128 KB shared memory per SM (99 KB per block usable), 32 MB L2. That
  is enough for the IQ3_S codebook (~2 KB) to live in smem + the K
  tile for GEMV.
- 16 GB VRAM. The whole hybrid SSM+MoE 35B-A3B IQ3_S model fits with
  `ncmoe=4` (first 4 layers' MoE experts offloaded to CPU), leaving
  568 MB free for KV cache at 32K `q8_0/q4_0`. The Qwen3.6 UD-IQ3_S
  takes 12.73 GiB and leaves significantly more headroom.
- CUDA 12.8 + driver 590.48 required for `sm_120`. `faster-whisper` /
  CTranslate2 are incompatible with `sm_120`, `whisper.cpp` large-v3
  turbo is used instead.

### Where it is limited

- **Memory bandwidth.** 448 GB/s peak (128-bit bus @ 28 Gbps). At M=1
  we hit 55% utilisation (246 GB/s), at M=4 we drop to 23% because of
  the GDN dispatch issue — not a compute ceiling, a scheduling one.
- **FFN/MoE CPU offload with `ncmoe`** is RAM-bandwidth bound on the
  DDR5-7200. Each `ncmoe=1` increment adds about 1 GB of tensor reads
  per token. `ncmoe=4` is sustainable, `ncmoe=8` tangibly slower.
- **Speculative decoding.** The ncmoe CPU offload makes `eval(16)` cost
  ≈ 16× the transfer of `eval(1)`. Even 100% acceptance with a 16-token
  draft yields zero speedup. MTP-1 measured at 47 tok/s vs 93 baseline.
  DFlash projected at 76 tok/s vs 93 baseline. Both are worse than pure
  AR on this hardware. (This is the crux of Pivot 2.)

GMKtec NucBox EVO-X2 — Ryzen AI Max+ 395 / gfx1151 / 96 GB LPDDR5X-8000
unified / 2599 €. Not as a production upgrade. As a **contribution

## 6. Lessons

1. **Agents will over-estimate speedups from techniques they do not
   own.** DFlash got pushed back into the conversation repeatedly by
   research agents who had not read the GDN State Barrier note. MTP
   same. Before accepting any "3-5× speedup" claim on this stack, the
   agent has to restate the GDN serialization constraint back to me in
   writing. Otherwise it is reading a paper, not reading our hardware.

2. **The sandboxing layer is load-bearing and occasionally wrong.**
   The scheduling gap analysis agent could not `Write` its output; the
   session had to relay the content. Two quick lessons: always have
   the host session able to write what the agent investigates; and
   don't mix permission debugging with investigation — separate runs.

3. **Opening the repos later than necessary cost nothing.** The first
   public commit is March 29, 2026, but the code was at prod-v1 level
   since March 21 and the runtime was running inside OpenClaw since
   early January. The ~3-month private iteration phase is where the
   hardest bugs died (seven in one day, March 17). Publishing before
   Q1 would have shipped a garbage runtime.

4. **The real bugs compound across layers.** Every catastrophic
   symptom I chased this quarter ("usercontentusercontentu...",
   Nemotron-H gibberish, greedy divergence) had the same shape: a
   small error per-layer that looks statistically negligible until
   you stack forty of them. The fix for each was one line — but
   finding each took a day because I looked at the wrong abstraction
   level. **Side-by-side layer-wise eval-callback traces first, code
   review second.**

5. **Peak tok/s is a vanity metric past a certain point.** Going from
   12 to 94 tok/s matters. Going from 94 to 100 does not change how
   the product feels. What changes the product is: TTFT at M=4,
   cross-slot contamination, reconnect behaviour on client drop,
   `finish_reason` actually being `"stop"`, Prometheus `/metrics`
   being honest. The April 24 session was worth more to users than
   any of the March perf sprints.

6. **The sensible quant pipeline is Unsloth + a custom mix on top,
   not a bespoke pipeline from scratch.** RAMP's value is the mix,
   not the C code. `llama-quantize --custom-q --imatrix` does 100%
   of what `ramp_quantize.c` did, in two commands instead of twelve.
   `ramp_quantize.c` is obsolete, not by choice but because the stock
   tool caught up. The only place to spend effort is the imatrix
   calibration corpus and the tensor-by-tensor mix decisions.

7. **Upstream quants of a maintainer's own family are the only fair
   baseline.** The TQ3 PR should never have gone out without first
   running IQ2_KS+IQ3_KS at the same size. Generic llama.cpp IQ3_S is
   not the baseline to beat when you're contributing to Ikawrakow's
   fork.

8. **`NativeScheduler` was not a rewrite, it was a parallel path.**
   The M1 multi-slot work did not touch the legacy HTTP generate path
   at all. It ran behind an env flag (`CHIMERE_MULTISLOT_NATIVE=1`)
   for the whole eight-jalon sprint. That meant we could iterate on
   it in prod without risking the running service. Every feature
   path going forward should have this property.

9. **Branches are free, long-running branches are not.** Fourteen
   feature branches alive on `origin` on April 24. Some (`m1-multislot-j1`)
   were healthy parallel work, merged cleanly. Some
   (`polish-prometheus`, `tq3-gguf-loader-support`) are diverged from
   `main` in non-trivial ways. The sensible policy, going forward,
   is: one feature branch per sprint jalon set, merge within 48h, and
   rebase as soon as `main` moves.

10. **Honesty in benchmarks compounds.** The stress test report
    documents a `finish_reason` bug (never `"stop"`, sampler ignores
    `<|im_end|>`) that was discovered while running the benchmark —
    and it is in the commit message alongside the headlines. The
    M=8/PCH=2048 92.9 tok/s cell is explicitly hedged as "N=12 too
    small to disambiguate". Hiding those would have made the report
    look cleaner; publishing them makes the next round of readers
    trust the rest.

## Closing note

Two months of public commits, preceded by two months of private work,
on a single-person weekend+evening budget. The single largest jump in
raw throughput (12 → 94 tok/s) happened before the repos opened. The
single largest jump in *product honesty* (fairness, TTFT under load,
Prometheus, reproducible benchmarks, published scheduling-gap analysis)
happened in the 24 hours of April 24.

None of this is novel in the "SOTA paper" sense. The novelty, if there
is any, is in the *combination*: a 35B hybrid SSM+MoE model running
fair multi-slot on a 16 GB consumer GPU, in Rust, with a
tokenizer-specific n-gram personalisation bias, against a backend that
we also contribute to. The GDN State Barrier is the one genuinely novel
theoretical contribution; it is also the reason half the speedup
techniques in the literature do not apply.

The runtime was 12 tok/s in January. It is 94 tok/s today. The operator
story is clearer, the benchmarks are reproducible, the scheduling
bottleneck is identified with exact line numbers, the fix is scoped to
three weeks. That is the ground state going into the next cycle.

— Kévin Rémondière, 2026-04-24, one long day.
