\begin{abstract}
We report a 27-day engineering effort to build a block diffusion drafter for speculative decoding on Qwen3.5-35B-A3B, a hybrid Gated DeltaNet (GDN) + Mixture-of-Experts model, using a single RTX~5060~Ti (16\,GB VRAM). We developed 8 drafter architectures, 7~C++ feature extractors, and ran 20~nightly training pipelines. Our best offline result is $\tau{=}6.06$ on a held-out set (v8 deep KV injection), comparable to the original DFlash ($\tau{\approx}6.4$). However, \textbf{wall-clock speedup is 0.73$\times$} (a slowdown, not an acceleration) because the target verification cost dominates on consumer hardware. More importantly, we identify a fundamental \emph{GDN State Barrier}: the recurrent state in GDN layers cannot be rolled back after draft rejection, making online speculative decoding structurally impossible on hybrid SSM-attention architectures without explicit state management. We document every failure, pivot, and lesson learned, and release all code and data.
\end{abstract}

\section{Introduction}

Speculative decoding~\cite{leviathan2023fast} accelerates autoregressive inference by having a fast drafter propose $K$ tokens that a target model verifies in parallel. DFlash~\cite{dflash2024} introduced block diffusion as the drafter mechanism, reporting $\tau{\approx}6.4$ tokens accepted per step on a dense target.

We attempted to bring DFlash to consumer hardware with a Mixture-of-Experts target (Qwen3.5-35B-A3B, 256~experts, 3.5B active/35B total). This paper is an honest account of what worked, what failed, and what we learned. The story is not one of steady improvement but of pivots, dead ends, and one structural impossibility that affects all hybrid SSM-attention models.

\textbf{Contributions:}
\begin{enumerate}
  \item A systematic exploration of 8 drafter architectures with complete training logs and benchmarks (\S\ref{sec:arch}).
  \item The formalization of the \emph{GDN State Barrier}, preventing online speculative decoding on hybrid architectures (\S\ref{sec:barrier}).
  \item An honest wall-clock analysis showing that speculative decoding \emph{slows down} inference on consumer MoE hardware (\S\ref{sec:wallclock}).
  \item All code, trained models, and 72{,}908 training blocks released openly.
\end{enumerate}

\section{Setup}

\textbf{Target model:} Qwen3.5-35B-A3B at IQ3\_S quantization (14.71\,GB, 3.56~BPW) on RTX~5060~Ti 16\,GB, running at 93~tok/s via ik\_llama.cpp sm\_120.

\textbf{Training data:} 3{,}927 diverse prompts (code, math, French clinical, general) $\to$ C++ feature extraction $\to$ 72{,}908 blocks of 16~tokens with hidden states from 5~intermediate layers (1, 10, 19, 28, 37).

\textbf{C++ extractors:} 7~versions over 5~days, evolving from full-sequence float32 (80\,MB/sample, 240\,GB total) to async-batched float16 with daemon protocol (20\,KB/sample). A resident target daemon with 12~binary commands (tokenize, eval, trim KV, save/restore GDN state) reduced reload overhead. Total extracted: 378K+ feature files, 226\,GB.

\subsection{v1: Gaussian Diffusion (Feb 28) --- Wrong Problem}

The first attempt used continuous Gaussian diffusion on token embeddings. Qwen3.5's embedding space has $\text{std}{=}0.0126$ --- adding Gaussian noise and predicting the original embedding cannot distinguish neighbors in a 248K vocabulary. Result: $\tau{=}0.62\%$.

\textbf{Lesson:} Predict token IDs via cross-entropy, not continuous embeddings.

\subsection{v3: Single-Shot Mask-Predict (Feb 28) --- Misleading Success}

An overnight pipeline produced the most impressive-looking result: $\tau{=}9.4$ (55.84\% acceptance) on 72{,}908 blocks, with 27.6\% perfect blocks (15/15 accepted). Three innovations: single-shot training from fully masked input, exponential position-weighted loss ($\gamma{=}7.0$), and anchor token conditioning.

\textbf{The problem:} This benchmark evaluated the training set. The singleshot benchmark on the same data with $n_\text{steps}{=}1$ shows acceptance stabilizing at $\sim$11.4\% --- still good, but $5{\times}$ lower than the headline number. We did not have a proper holdout set for v3.

\textbf{Lesson:} Always benchmark on held-out data. Train-set $\tau$ is meaningless for deployment.

\subsection{v5: Full-Context KV Injection (Mar 1) --- Infrastructure Battles}

The architecturally correct approach: each drafter layer cross-attends to the full target context, not just a fused anchor vector. 295M trainable + 1{,}017M frozen Qwen weights (embed\_tokens + lm\_head).

Four sub-variants were tried: v5-daemon (NaN-corrupted features, val\_acc=32.8\%), v5-qwen (shared weight mismatch, val\_acc=12.2\%), v5-residual (marginal, val\_acc=13.7\%), v5-fullmask (too hard, val\_acc=17.9\%). All had 702/2{,}855 samples with NaN/Inf in extracted features. Two training runs crashed with OOM on checkpoint resume:

\begin{verbatim}
torch.OutOfMemoryError: Tried to allocate 970 MiB.
GPU 0 has 15.47 GiB total, 173.56 MiB free.
\end{verbatim}

\textbf{Lesson:} Feature extraction quality matters more than drafter architecture. The quant mismatch (IQ3\_S hidden states $\neq$ BF16 lm\_head) is a real problem for shared-weight drafters.

\subsection{v6: z-lab Alignment (Mar 2) --- Correct but Weak}

Full alignment with the DFlash paper: 8~layers, GQA (32/4 heads), unified KV injection, streak distillation loss ($\gamma{=}7.0$). Trained on up to 98K samples with torch.compile ($1{,}422$~tok/s, $+58\%$).

Result: $\tau{=}2.71\%$ --- 60\% of blocks completely rejected, 40\% accept exactly 1~token. The drafter learns position~1 but cannot predict further. Without positional encoding, all 16 masked positions receive identical representations.

\subsection{v7: RoPE --- The False Victory (Mar 4)}

Adding RoPE ($\theta{=}10^7$) improved val\_loss from 5.06 to 4.02 and val\_acc from 8.3\% to 26.2\%. But the benchmark produced \emph{two contradictory results}: $\tau{=}16.96\%$ with correct absolute positions and $\tau{=}0.25\%$ with relative positions (a bug).

\textbf{Critical insight:} \emph{Better validation loss does not predict better $\tau$.} v7 has the best val\_loss among discrete versions (4.02) but the worst $\tau$ (0.25\%) when positions are wrong. The auto-generated report used the wrong benchmark, concluding ``v7 is worse than v6.''

\subsection{v8: Deep KV Injection (Mar 4) --- The Holdout Breakthrough}

Per-layer KV projections (+67M params) give each drafter layer fresh target signal. Combined with LK loss ($\eta{=}3.0$) for direct acceptance optimization. Trained on 30K holdout-separated samples.

Result: $\tau_{\text{holdout}}{=}6.06\%$ ($\sim$0.9 tokens accepted per block), estimated speedup $1.63{\times}$. This is the first result measured on a proper holdout set, and it is comparable to DFlash's $\tau{\approx}6.4$ on a different target.

\subsection{EAGLE: Autoregressive Attempt (Mar 4)}

A pivot to autoregressive drafting (1-layer, double residual, causal attention). Abandoned because sequential token generation in the block defeats the purpose of block diffusion --- it cannot draft 16~tokens in parallel.

\section{The GDN State Barrier}
\label{sec:barrier}

Despite offline success, we could not deploy online speculative decoding. Three online adaptation cycles were attempted (with 5{,}600+ captured samples), and one cycle achieved $\tau{=}99\%$ on its own buffer --- a clear sign of catastrophic overfitting.

The fundamental problem: Qwen3.5-35B-A3B has 30~GDN recurrent layers and 10~attention layers. In speculative decoding, rejected draft tokens must be ``undone'' from the target's state. For attention layers, this means trimming the KV cache (position-indexed, $O(1)$). For GDN layers:
\begin{equation}
  M_n = \alpha_n \cdot M_{n-1} \cdot (I - k_n q_n^\top) + k_n v_n^\top
\end{equation}
$M_n$ is a lossy compression of all prior tokens. There is no inverse operation. Calling \texttt{llama\_memory\_seq\_rm} on GDN state is undefined behavior.

\textbf{Four structural walls in llama.cpp:}
\begin{enumerate}
  \item No \texttt{seq\_rm} for recurrent layers.
  \item No state snapshot/restore API (though our C++ daemon implements \texttt{CMD\_STATE\_SAVE\_GDN}/\texttt{CMD\_STATE\_RESTORE\_GDN}).
  \item No batch verification with rollback.
  \item Draft and target share the same recurrent state.
\end{enumerate}

This affects all hybrid architectures: Jamba (Mamba + attention), RWKV-6/7, Qwen3.5, and any future model mixing recurrent and attention layers.

\section{Wall-Clock Reality}
\label{sec:wallclock}

The headline numbers hide a painful truth:

\begin{table}[h]
\centering
\caption{Wall-clock analysis (v8, offline setting).}
\begin{tabular}{lr}
\toprule
Target forward (IQ3\_S, ik\_llama) & $\sim$11\,ms/token \\
Draft block (v8, 16 tokens) & 2.7\,ms \\
Tokens accepted per block ($\tau_\text{holdout}$) & 0.9 \\
Time per accepted token (spec decode) & $(11 + 2.7) / 0.9 = 15.2$\,ms \\
Time per token (pure AR) & 11\,ms \\
\textbf{Wall-clock speedup} & \textbf{0.73$\times$ (slowdown)} \\
\bottomrule
\end{tabular}
\end{table}

At $\tau{=}0.9$ accepted tokens per block, the drafter overhead \emph{exceeds} the savings from parallel verification. Speculative decoding on this hardware configuration requires $\tau{>}2.0$ to break even.

\textbf{Why:} On consumer hardware with CPU expert offloading (\texttt{-ncmoe 4}), the target is already fast (93~tok/s, 11~ms/token). Each draft + verify cycle costs $13.7$\,ms but only produces $0.9$ tokens on average. On a slower target (e.g., A100 with batch=32, 50+ms/token), the same drafter would be profitable.

\section{What We Tried That Failed}

\begin{itemize}
  \item \textbf{Continuous Gaussian diffusion} (v1): $\tau{=}0.62\%$. Wrong problem formulation.
  \item \textbf{Shared Qwen weights} (v5-qwen): quant mismatch degraded accuracy by $4{\times}$.
  \item \textbf{Training without positional encoding} (v6): 60\% complete rejection.
  \item \textbf{Wrong position encoding} (v7 benchmark): $70{\times}$ discrepancy in $\tau$ from a position bug.
  \item \textbf{Online adaptation}: $\tau{=}99\%$ on buffer = catastrophic overfitting in 3~cycles.
  \item \textbf{EAGLE autoregressive}: sequential generation defeats parallel drafting.
  \item \textbf{Feature extraction v1}: 80\,MB/sample $\times$ 3{,}000 = 240\,GB. 702 samples with NaN.
\end{itemize}

\section{What We Learned}

\begin{enumerate}
  \item \textbf{Train-set $\tau$ is useless.} v3's $\tau{=}9.4$ (train) became $\sim$11\% (realtime) became 6.06\% (holdout). Report holdout $\tau$ only.
  \item \textbf{Val loss $\neq$ $\tau$.} v7 has the best val\_loss but the worst $\tau$ (with wrong positions).
  \item \textbf{The GDN barrier is structural.} No amount of drafter improvement fixes the inability to roll back recurrent state. This requires a new runtime.
  \item \textbf{Consumer MoE is too fast for spec decode.} At 93~tok/s, the drafter overhead dominates. Spec decode is for slow targets.
  \item \textbf{Feature quality $>$ drafter architecture.} NaN features, quant mismatches, and extraction bugs caused more failures than architectural choices.
  \item \textbf{Seven C++ extractors in 5 days} is the real engineering cost. The drafter training takes 66~minutes; building reliable feature extraction took weeks.
  \item \textbf{Nightly pipelines fail 50\% of the time.} Of 20~nightly runs, $\sim$10 completed successfully. Server conflicts, VRAM exhaustion, and extraction crashes were routine.
\end{enumerate}

\section{Related Work}

DFlash~\cite{dflash2024} introduced block diffusion drafting. EAGLE~\cite{li2024eagle} and Medusa~\cite{cai2024medusa} propose learned drafters for attention-only models. SpecMamba~\cite{zhang2024specmamba} identified SSM speculation challenges but did not address hybrid architectures. BD3-LMs~\cite{arriaga2024block} introduced block diffusion for language modeling.

\section{Conclusion}

We spent 27~days building a block diffusion drafter for a hybrid MoE model on consumer hardware. The technical result ($\tau{=}6.06$ holdout, comparable to DFlash) is secondary to the structural findings: the GDN State Barrier prevents online deployment, and consumer MoE models are too fast for speculative decoding to help.

These are results the community needs before investing in speculative decoding for hybrid architectures. We release everything: 8~architectures, 7~extractors, 20~nightly logs, 72{,}908~training blocks, and 56K~lines of Rust runtime code that bypasses the barrier by design.
