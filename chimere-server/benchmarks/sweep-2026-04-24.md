## 2. Methodology

For each cell of the grid:

1. Launch a fresh chimere-server child process on the bench port (not :8081 prod). Wait for /health (up to 180 s, model load is 30-60 s).
2. Snapshot /metrics and /v1/status pre-pass.
3. Launch `nvidia-smi dmon -s pumct -d 1 -c 1200 -o T` concurrently.
4. Drive 12 concurrent streaming requests through the existing stream_bench.py driver (TTFT = arrival of first SSE delta with non-empty content or reasoning_content, inter-token gap = wall-time between consecutive deltas).
5. Snapshot /metrics and /v1/status post-pass.
6. Stop the server (SIGTERM, 20 s grace, then SIGKILL).
7. Wait 3 s for GPU VRAM release before the next cell.

Aggregations are computed from the driver's per-request JSONL (raw-<cell>.jsonl) and the nvidia-smi dmon CSV. Quantiles are computed on the pooled sample (all requests x all tokens for inter-token distributions).

VRAM column is a single-shot `nvidia-smi` query at end-of-pass, so MiB used reflects model weights + KV cache + activation pools at the moment the pass finished. Good enough for sweet-spot ranking, not publication-grade.

## 8. Caveats

- Small N per cell. At 12 requests/cell, TTFT p99 has sample size N. Good enough for ranking; not publication-grade. Bump --n-requests-per-pass for tighter error bars.
- One GPU, one model, one quant. Generalization to other quants or hybrids not measured by this sweep.
- Cold-start dominates for short-lived cells. The 30-60 s model load per cell is not included in per-cell wall_s (the pass clock only starts after /health returns). The harness itself takes ~60-90 s per cell end-to-end.
- VRAM measurement is single-shot nvidia-smi at pass end, no peak tracking. The dmon CSV has BW-util (%), not MiB. Use the existing run-bench.sh + external VRAM monitoring if a peak is required.
- Native streaming only. At CHIMERE_MULTISLOT_NATIVE=1, non-streaming requests return HTTP 400 (native_mode_streaming_only). The driver uses streaming exclusively.
